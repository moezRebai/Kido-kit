Eight stages, one feature, real output

Every command in the Kido pipeline, in the order they actually run, showing exactly what each one produces — the files it writes, the questions it asks, the Jira and git actions it takes. BA has no git push access, so Jira — not git — is what carries the spec from BA to Dev.

Worked example throughout: pricing-service, an existing (brownfield) microservice, gaining a new capability — Swap Pricing against the current curve model.

2
/kido:specify one session · functional-spec.md + design.md

Define what to build, and how — in one sitting

One run of /kido:specify walks straight through both specs back to back — BA doesn't need Dev in the room for this. Functional pass first: brainstorming-style grill, one question at a time, reading functional-docs.md first so nothing contradicts what's already documented. Then, same session, the design pass — if Dev genuinely isn't present, the agent drafts design.md itself, flagged for Dev to validate later, not a blocking requirement. Both passes require enumerating boundary/failure/concurrent-access conditions before being considered done.

Push both to Jira as one Epic?
Files written
kido/changes/add-swap-pricing/functional-spec.mdjiraId: PROJ-501
# Swap Pricing

Traders need to price interest-rate swaps against the live curve, not just vanilla instruments.

## Acceptance criteria
- Given a swap definition, when priced, then return PV using the current curve
- Given an invalid notional, then reject with a clear error

## Boundary / failure / concurrency
- Curve unavailable mid-price: reject with a retryable error, no partial PV returned
- Two requests pricing the same swap concurrently: independent, no shared mutable state

## Out of scope
- Cross-currency swaps (separate capability, later)
kido/changes/add-swap-pricing/design.md
# Swap Pricing — Technical Design

## Approach
Extend the existing calculator module with a curve-based swap pricer, reusing the immutable curve value object rather than introducing a new pricing engine.

## Data model
New SwapDefinition DTO — notional, fixed rate, tenor, currency

## Testing strategy
Unit tests per cash-flow leg; integration test against a fixture curve

## Boundary / failure / concurrency
Curve lookup failure surfaces as a typed exception, not a null — caller decides retry vs. fail

## ADR (new)
Swap PV computed leg-by-leg rather than via a closed-form formula — matches the existing vanilla pricer's pattern, keeps both pricers auditable the same way.

/kido:tasks BA-run · vertical slices · Jira Stories

Break it into shippable slices

Reads both functional-spec.md and design.md. Tasks are cut as vertical slices — each one crosses every layer it touches and can be verified on its own — never "all the schema changes" as one task and "all the API changes" as another. That's the end of BA's part — no branch gets created here; BA has no push access, so that's Dev's step next.

4
kido jira pull Dev's entry · Jira key → local files → branch

Dev picks it up — by Jira key, not git

BA's functional-spec.md/design.md never reached this machine via git — Jira was the transport. Dev gives /kido:apply a Jira key; since nothing's local yet, it runs kido jira pull first — resolving the Epic, reconstructing functional-spec.md + design.md from its description and tasks.md from every child Story. Only then, before any task starts, does it ask to create the branch.

/kido:apply one subagent per task · TDD

Implement, task by task

Each task gets its own subagent, scoped to just that task's context — spec, design, its one task, and kido/docs/. Test written first, confirmed failing, then made to pass. Task 2 waits for Task 1 since it depends on it.

/kido:review per task, automatic + on demand

Check it against what was promised

Runs after every task, not just once at the end — an explicit step this session takes, not something that fires on its own. Two checks, always both: does the diff match design.md/tasks.md, and does it meet the conventions technical-docs.md documents.

/kido:archive git + docs + Jira · always last

Close it out

Confirms review actually happened for every task before anything else. Then, in order: commit the code (never bundling in planning docs), optionally refresh kido/docs/, move the change to kido/changes/archive/ as its own separate commit, sync any local spec/task edits back to Jira, and mark Jira done.

One open question not yet resolved in the tool itself: if the code-commit PR merges before the archive-docs commit gets pushed, pushing to the same feature branch is risky (many teams auto-delete branches after merge) — reorder into one PR vs. detect merge state and commit to main directly are both still on the table.

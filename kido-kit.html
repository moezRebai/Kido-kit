Overview

What Kido is, and how it's used

Kido is a spec-driven collaboration workflow for a BA and a Dev working through AI agents on a shared unit of work — inspired by OpenSpec's proposal → design → tasks → apply → archive pipeline and mattpocock/skills' library of independent team disciplines. It works across greenfield (nothing exists yet) and brownfield (existing codebase to reconcile with) projects, one microservice repo at a time, and ships as an installable npm CLI (kido) plus generated Claude Code skills/commands — not a one-off script or a manual checklist.

Who does what

RoleResponsibility
BAWrites functional specs, breaks them into Jira Epics/Stories. Has no git push access — Jira, not git, is the handoff to Dev.
DevWrites technical specs, implements against the spec with TDD, code-reviews, pushes to git and opens the PR.

What gets scaffolded in the repo

Running kido init once per microservice repo creates:

tree
<microservice-repo>/
├── kido/
│   ├── docs/
│   │   ├── {project}-functional-docs.md   # BA-facing
│   │   └── {project}-technical-docs.md    # Dev-facing
│   └── changes/
│       ├── <change-name>/                # one per feature/bug in flight
│       └── archive/                       # moved here by /kido:archive
└── .claude/
    ├── skills/mr-*/SKILL.md               # orchestration logic
    └── commands/kido/*.md                 # /kido:* commands

kido/changes/ (once archived) is committed to git — the durable record of why the codebase looks the way it does. .claude/ stays untracked; it's fully regeneratable via kido init.

CLI commands (data-ops only)

CommandBehavior
kido initScaffolds kido/docs + kido/changes, generates skills/commands.
kido new-changeScaffolds kido/changes/<name> with the right starter files.
kido statusReports which artifacts exist/are complete for a change.
kido validateChecks required artifacts are present before archiving.
kido archiveMoves a change folder into changes/archive/.
kido jira syncPushes spec/tasks/bug content to Jira — idempotent create-or-update.
kido jira pullReverse of sync — materializes a whole change from any Jira key.
kido docs exportCopies kido/docs/* into another repo — cross-repo seeding.

Generated skills/commands — the actual pipeline

CommandBehavior
/kido:documentBrownfield-only 5-phase discovery; also refreshes existing docs any time.
/kido:specifyBA's entry point — bug/feature fork, functional-spec.md, a checkpoint (push? new or existing Epic?), then design.md.
/kido:tasksBA-run, vertical-slice task breakdown — new-Epic mode only.
/kido:applyDev's entry point — pulls from Jira if needed, creates the branch, implements with TDD.
/kido:reviewSpec-traceability + standards — runs per-task and on demand.
/kido:archiveCommit, optionally refresh docs, archive the change, sync Jira — always last.

Jira hierarchy

functional-spec.md becomes its own Epic by default, or a single Story nested under an existing Epic for small, single-unit features. tasks.md tasks become Stories directly under the Epic. bug.md becomes a Bug ticket, same tracker, distinct type. Every sync point — either direction — asks first, never silent.

Out of scope for v1

  • Gemini CLI / Kilo Code renderers — Claude Code only for now (the architecture supports adding them later).
  • Cross-microservice functional specs — one store per microservice, owner-microservice model only.
  • Same-session BA+Dev live collaboration — async handoff only, Jira is the transport.
  • Bidirectional/continuous Jira sync — kido jira pull is a one-shot materialize, not an ongoing two-way sync.

/kido:specify one command, two forks

Where the three workflows actually split

Everything up to here is identical no matter what kind of work this is. /kido:specify is BA's single entry point — it forks twice: first bug vs. feature, then, for features, how much Jira structure this needs.

Fork 1 — bug or feature?
Bug

Grilled with the same rigor as a functional spec — description, repro steps, boundary/failure/concurrency conditions — but produces one document, full stop.

kido/changes/stale-quote-on-reconnect/bug.mdjiraId: PROJ-610
# Stale quote survives a SignalR reconnect

Repro: kill the connection mid-session, reconnect — the last quote before the drop keeps rendering until a new tick arrives.

## Boundary / failure / concurrency
- Affects every open session on that instrument, not just the one that dropped
- Race: a tick arriving during reconnect must win over the stale cached value

Feature

Continues below — functional-spec.md gets written, then the second fork happens.

Feature path — functional-spec.md
kido/changes/add-swap-pricing/functional-spec.mdjiraId: PROJ-501
# Swap Pricing

Traders need to price interest-rate swaps against the live curve, not just vanilla instruments.

## Acceptance criteria
- Given a swap definition, when priced, then return PV using the current curve
- Given an invalid notional, then reject with a clear error

## Boundary / failure / concurrency
- Curve unavailable mid-price: reject with a retryable error, no partial PV returned
- Two requests pricing the same swap concurrently: independent, no shared mutable state

## Out of scope
- Cross-currency swaps (separate capability, later)
Fork 2 — new Epic, or does this belong under one that already exists?

Reached once functional-spec.md is written — the checkpoint stops here rather than sliding silently into design. It asks whether to push now, whether this needs a new Epic or an existing one, and whether to continue into design grilling now or wait for Dev.

New Epicdefault

The normal case — functional-spec.md + design.md become their own Epic. Continues to Stage 3 — /kido:tasks for a full task breakdown.

kido/changes/add-swap-pricing/design.md
# Swap Pricing — Technical Design

## Approach
Extend the existing calculator module with a curve-based swap pricer, reusing the immutable curve value object rather than introducing a new pricing engine.

## Data model
New SwapDefinition DTO — notional, fixed rate, tenor, currency

## Testing strategy
Unit tests per cash-flow leg; integration test against a fixture curve

## Boundary / failure / concurrency
Curve lookup failure surfaces as a typed exception, not a null — caller decides retry vs. fail

## ADR (new)
Swap PV computed leg-by-leg rather than via a closed-form formula — matches the existing vanilla pricer's pattern, keeps both pricers auditable the same way.

Dev picks it up — by Jira key, not git

.claude/ never gets committed here — it's fully regenerated by kido init, so there's nothing to preserve.

Reflects the pipeline as currently implemented — every file, prompt, and command shown here is real, not illustrative. A fourth shape — a feature under an existing Epic that does need multiple tasks — isn't supported yet: kido jira pull would have no way to tell that feature's Stories apart from unrelated ones sharing the same Epic. /kido:specify steers that case toward "new Epic" instead rather than producing something it can't reconstruct correctly. Separately, one open question on the git side: if the code-commit PR merges before the archive-docs commit gets pushed, pushing to the same feature branch is risky (many teams auto-delete branches after merge) — reorder into one PR vs. detect merge state and commit to main directly are both still on the table.
